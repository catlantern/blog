## 一、引言：为什么需要IPsec

### 1.1 网络安全威胁

在传统IP网络中，数据包以明文传输，存在诸多安全威胁：

```
┌─────────────────────────────────────────────────────────────┐
│                  网络安全威胁                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  威胁1：窃听（Eavesdropping）                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 数据包明文传输，可被截获                           │   │
│  │  • 敏感信息泄露（密码、信用卡号等）                   │   │
│  │  • 商业机密被盗                                      │   │
│  │                                                          │   │
│  │  示例：Wi-Fi窃听、网络嗅探                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  威胁2：篡改（Tampering）                                     │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 数据包内容可被修改                                 │   │
│  │  • 破坏数据完整性                                    │   │
│  │  • 注入恶意代码                                      │   │
│  │                                                          │   │
│  │  示例：中间人攻击、DNS劫持                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  威胁3：伪造（Spoofing）                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 伪造源IP地址                                       │   │
│  │  • 冒充合法用户                                      │   │
│  │  • 绕过访问控制                                      │   │
│  │                                                          │   │
│  │  示例：IP欺骗、ARP欺骗                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  威胁4：重放攻击（Replay Attack）                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 截获并重发旧数据包                                 │   │
│  │  • 绕过认证机制                                      │   │
│  │  • 造成重复操作                                      │   │
│  │                                                          │   │
│  │  示例：重放认证包、重放交易请求                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、IPsec协议基础

### 2.1 IPsec协议组成

IPsec是一套协议族，包含多个协议：

```
┌─────────────────────────────────────────────────────────────┐
│                    IPsec协议组成                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. AH（Authentication Header）                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  功能：                                              │   │
│  │  • 提供数据完整性和认证                               │   │
│  │  • 不提供加密                                        │   │
│  │  • 协议号：51                                        │   │
│  │                                                          │   │
│  │  特点：                                              │   │
│  │  • 保护IP头（除了可变字段）                           │   │
│  │  • 使用HMAC算法                                      │   │
│  │  • 不支持NAT穿越                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  2. ESP（Encapsulating Security Payload）                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  功能：                                              │   │
│  │  • 提供加密、完整性和认证                             │   │
│  │  • 最常用的IPsec协议                                 │   │
│  │  • 协议号：50                                        │   │
│  │                                                          │   │
│  │  特点：                                              │   │
│  │  • 加密载荷                                          │   │
│  │  • 支持多种加密算法                                   │   │
│  │  • 支持NAT穿越                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  3. IKE（Internet Key Exchange）                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  功能：                                              │   │
│  │  • 自动协商SA                                        │   │
│  │  • 密钥管理和分发                                     │   │
│  │  • 认证对等体                                        │   │
│  │                                                          │   │
│  │  版本：                                              │   │
│  │  • IKEv1：RFC 2409                                   │   │
│  │  • IKEv2：RFC 7296（推荐）                           │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  4. IPcomp（IP Payload Compression）                         │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  功能：                                              │   │
│  │  • 压缩IP载荷                                        │   │
│  │  • 减少带宽占用                                      │   │
│  │  • 协议号：108                                       │   │
│  │                                                          │   │
│  │  算：通常与ESP配合使用                               │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 IPsec工作模式

```
┌─────────────────────────────────────────────────────────────┐
│                    IPsec工作模式                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 传输模式（Transport Mode）                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  特点：                                              │   │
│  │  • 只保护载荷，不保护IP头                             │   │
│  │  • 端到端通信                                        │   │
│  │  • 不改变IP头                                        │   │
│  │                                                          │   │
│  │  报文结构：                                          │   │
│  │  [IP Header] [AH/ESP Header] [Payload] [ICV]         │   │
│  │                                                          │   │
│  │  应用场景：                                          │   │
│  │  • 端到端通信                                        │   │
│  │  • 服务器间通信                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  2. 隧道模式（Tunnel Mode）                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  特点：                                              │   │
│  │  • 保护整个IP包                                      │   │
│  │  • 添加新的IP头                                      │   │
│  │  • VPN场景                                           │   │
│  │                                                          │   │
│  │  报文结构：                                          │   │
│  │  [New IP Header] [AH/ESP Header]                     │   │
│  │  [Original IP Header] [Payload] [ICV]                │   │
│  │                                                          │   │
│  │  应用场景：                                          │   │
│  │  • Site-to-Site VPN                                  │   │
│  │  • Remote Access VPN                                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  模式对比：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  特性            传输模式        隧道模式              │   │
│  │  ────────────────────────────────────────────────   │   │
│  │  保护范围        仅载荷          整个IP包             │   │
│  │  IP头            不变            新增外层IP头         │   │
│  │  开销            小              大                   │   │
│  │  应用场景        端到端          VPN                  │   │
│  │  NAT兼容         困难            容易                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

SA是IPsec的核心概念：

```
┌─────────────────────────────────────────────────────────────┐
│                    SA（Security Association）                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  定义：                                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  SA是两个通信实体之间单向的安全关联，                  │   │
│  │  定义了如何保护数据（加密算法、密钥、模式等）。        │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  SA三元组标识：                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • SPI（Security Parameter Index）：32位标识符       │   │
│  │  • 目的IP地址                                        │   │
│  │  • 安全协议（AH或ESP）                               │   │
│  │                                                          │   │
│  │  示例：(SPI=0x12345678, Dst=192.168.1.2, ESP)        │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  SA包含的信息：                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 加密算法和密钥（ESP）                             │   │
│  │  • 认证算法和密钥（AH/ESP）                          │   │
│  │  • 工作模式（传输/隧道）                              │   │
│  │  • 序列号（抗重放）                                  │   │
│  │  • 生存时间                                          │   │
│  │  • 源/目的IP地址                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  SA特点：                                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 单向：一个SA只能保护一个方向的数据                 │   │
│  │  • 双向通信需要两个SA（入站和出站）                   │   │
│  │  • 存储在SAD（Security Association Database）        │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.4 SP（Security Policy）

SP定义了哪些流量需要IPsec保护：

```
┌─────────────────────────────────────────────────────────────┐
│                    SP（Security Policy）                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  定义：                                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  SP定义了哪些流量需要IPsec保护，以及如何保护。        │   │
│  │  SP存储在SPD（Security Policy Database）。            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  SP包含的信息：                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 选择器（Selector）：匹配流量的规则                 │   │
│  │    - 源/目的IP地址                                   │   │
│  │    - 源/目的端口                                     │   │
│  │    - 协议                                            │   │
│  │  • 动作（Action）：                                  │   │
│  │    - discard：丢弃                                   │   │
│  │    - bypass：绕过IPsec                               │   │
│  │    - apply：应用IPsec                                │   │
│  │  • 优先级：多条SP时的匹配顺序                         │   │
│  │  • 方向：入站或出站                                  │   │
│  │  • SA模板：如何创建SA                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  SP匹配流程：                                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  1. 根据流量特征（IP、端口、协议）查找SPD             │   │
│  │  2. 按优先级匹配SP                                   │   │
│  │  3. 根据动作处理：                                   │   │
│  │     - discard：丢弃数据包                            │   │
│  │     - bypass：直接转发                               │   │
│  │     - apply：查找或创建SA，应用IPsec                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 三、XFRM核心数据结构

### 3.1 XFRM框架概述

XFRM（Transform）是Linux内核的IPsec实现框架：

```
┌─────────────────────────────────────────────────────────────┐
│                    XFRM框架概述                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  XFRM组件：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  1. xfrm_state：SA的实现                             │   │
│  │     • 定义单个SA的所有信息                           │   │
│  │     • 存储在内核的SAD中                              │   │
│  │                                                          │   │
│  │  2. xfrm_policy：SP的实现                             │   │
│  │     • 定义流量匹配规则和处理动作                      │   │
│  │     • 存储在内核的SPD中                               │   │
│  │                                                          │   │
│  │  3. xfrm_tmpl：SA模板                                 │   │
│  │     • 用于创建SA                                      │   │
│  │     • 包含在xfrm_policy中                             │   │
│  │                                                          │   │
│  │  4. xfrm_dst：路由缓存                                │   │
│  │     • 缓存IPsec路由信息                               │   │
│  │     • 避免重复查找                                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  XFRM处理流程：                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  出站流程：                                          │   │
│  │  1. 查找SP（xfrm_policy_lookup）                     │   │
│  │  2. 查找或创建SA（xfrm_state_lookup）                │   │
│  │  3. 应用IPsec处理（加密、认证）                       │   │
│  │  4. 发送数据包                                       │   │
│  │                                                          │   │
│  │  入站流程：                                          │   │
│  │  1. 根据SPI查找SA（xfrm_state_lookup）               │   │
│  │  2. 验证和解密                                        │   │
│  │  3. 查找SP验证策略                                   │   │
│  │  4. 送入协议栈                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 xfrm_state结构体

`xfrm_state`是SA的核心数据结构：

```c
// include/net/xfrm.h: 161
struct xfrm_state {
    possible_net_t      xs_net;        // 网络命名空间
    union {
        struct hlist_node  gclist;     // 垃圾回收链表
        struct hlist_node  bydst;      // 按目的地址哈希
    };
    struct hlist_node   bysrc;         // 按源地址哈希
    struct hlist_node   byspi;         // 按SPI哈希
    struct hlist_node   byseq;         // 按序列号哈希

    refcount_t          refcnt;        // 引用计数
    spinlock_t          lock;          // 自旋锁

    struct xfrm_id      id;            // SA标识（SPI、目的地址、协议）
    struct xfrm_selector sel;          // 选择器（匹配流量）
    struct xfrm_mark    mark;          // 标记
    u32                 if_id;         // 接口ID
    u32                 tfcpad;        // TFC填充

    u32                 genid;         // 生成ID

    /* Key manager bits */
    struct xfrm_state_walk  km;        // 密钥管理

    /* Parameters of this state. */
    struct {
        u32    reqid;                  // 请求ID
        u8     mode;                   // 模式（传输/隧道）
        u8     replay_window;          // 重放窗口大小
```

```c
// net/xfrm/xfrm_state.c: 1247
struct xfrm_state *xfrm_state_lookup_byspi(struct net *net, __be32 spi,
                                            unsigned short family)
{
    struct xfrm_state *x;
    struct xfrm_state_walk *w;

    spin_lock_bh(&net->xfrm.xfrm_state_lock);
    // 遍历所有SA
    list_for_each_entry(w, &net->xfrm.state_all, all) {
        x = container_of(w, struct xfrm_state, km);
        if (x->props.family != family || x->id.spi != spi)
            continue;

        xfrm_state_hold(x);  // 增加引用计数
        spin_unlock_bh(&net->xfrm.xfrm_state_lock);
        return x;
    }
    spin_unlock_bh(&net->xfrm.xfrm_state_lock);
    return NULL;
}
```

```c
// net/xfrm/xfrm_policy.c: 2099
static struct xfrm_policy *xfrm_policy_lookup_bytype(struct net *net, u8 type,
                                                      const struct flowi *fl,
                                                      u16 family, u8 dir,
                                                      u32 if_id)
{
    struct xfrm_pol_inexact_candidates cand;
    const xfrm_address_t *daddr, *saddr;
    struct xfrm_pol_inexact_bin *bin;
    struct xfrm_policy *pol, *ret;
    struct hlist_head *chain;
    unsigned int sequence;
    int err;

    daddr = xfrm_flowi_daddr(fl, family);
    saddr = xfrm_flowi_saddr(fl, family);
    if (unlikely(!daddr || !saddr))
        return NULL;

    rcu_read_lock();
 retry:
    do {
        sequence = read_seqcount_begin(&net->xfrm.xfrm_policy_hash_generation);
        // 步骤1：在哈希表中查找
        chain = policy_hash_direct(net, daddr, saddr, family, dir);
    } while (read_seqcount_retry(&net->xfrm.xfrm_policy_hash_generation, sequence));

    ret = NULL;
    // 步骤2：遍历哈希链表
    hlist_for_each_entry_rcu(pol, chain, bydst) {
        // 步骤3：匹配SP
        err = xfrm_policy_match(pol, fl, type, family, dir, if_id);
        if (err) {
            if (err == -ESRCH)
                continue;
            // ...
        }
        // 步骤4：检查优先级
        if (ret == NULL || ret->priority > pol->priority)
            ret = pol;
    }

    // 步骤5：查找不精确匹配
    // ...

    rcu_read_unlock();
    return ret;
}
```

```c
// net/xfrm/xfrm_output.c: 702
int xfrm_output(struct sock *sk, struct sk_buff *skb)
{
    struct net *net = dev_net(skb_dst(skb)->dev);
    struct xfrm_state *x = skb_dst(skb)->xfrm;
    int err;

    // 根据地址族初始化
    switch (x->outer_mode.family) {
    case AF_INET:
        memset(IPCB(skb), 0, sizeof(*IPCB(skb)));
        IPCB(skb)->flags |= IPSKB_XFRM_TRANSFORMED;
        break;
    case AF_INET6:
        memset(IP6CB(skb), 0, sizeof(*IP6CB(skb)));
        IP6CB(skb)->flags |= IP6SKB_XFRM_TRANSFORMED;
        break;
    }

    secpath_reset(skb);

    // 检查硬件卸载
    if (xfrm_dev_offload_ok(skb, x)) {
        struct sec_path *sp;

        sp = secpath_set(skb);
        if (!sp) {
            XFRM_INC_STATS(net, LINUX_MIB_XFRMOUTERROR);
            kfree_skb(skb);
            return -ENOMEM;
        }

        sp->olen++;
        sp->xvec[sp->len++] = x;
        xfrm_state_hold(x);

        if (x->xso.type == XFRM_DEV_OFFLOAD_ACQUIRE_IN_PROGRESS) {
            x->xso.type = XFRM_DEV_OFFLOAD_PACKET;
        }

        return xfrm_output_offload_ok(skb, x);
    }

    // 软件处理
    return xfrm_output2(sk, skb);
}
```

```c
    // 步骤6：计算ICV（完整性校验值）
```

```c
    aead_request_set_ad(req, assoclen);

    err = crypto_aead_decrypt(req);

    // 步骤5：验证ICV
    // ...

    // 步骤6：去除填充
    // ...

    // 步骤7：返回下一个协议
    return nexthdr;
}
```

```bash
    src 10.0.0.0/24 dst 10.0.1.0/24 \
    dir out \
    tmpl src 192.168.1.1 dst 192.168.1.2 \
    proto esp mode tunnel reqid 1

# 步骤2：配置SP（入站）
ip xfrm policy add \
    src 10.0.1.0/24 dst 10.0.0.0/24 \
    dir in \
    tmpl src 192.168.1.2 dst 192.168.1.1 \
    proto esp mode tunnel reqid 1

# 步骤3：配置SA（出站）
ip xfrm state add \
    src 192.168.1.1 dst 192.168.1.2 \
    proto esp spi 0x12345678 \
    mode tunnel reqid 1 \
    auth hmac-sha256 0x1234567890abcdef... \
    enc aes 0xabcdef1234567890...

# 步骤4：配置SA（入站）
ip xfrm state add \
    src 192.168.1.2 dst 192.168.1.1 \
    proto esp spi 0x87654321 \
    mode tunnel reqid 1 \
    auth hmac-sha256 0xfedcba0987654321... \
    enc aes 0x0987654321fedcba...

# 步骤5：配置路由
ip route add 10.0.1.0/24 via 192.168.1.2
```

```
│  ┌─────────────────────────────────────────────────────┐   │
```

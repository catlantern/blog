## 一、引言：为什么需要NUMA

### 1.1 一个真实的问题场景

在现代多核服务器上，经常会出现这样的诡异问题：

```
┌─────────────────────────────────────────────────────────────┐
│                    CPU越多，性能反而下降？                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  场景：32核服务器运行数据库                                  │
│                                                             │
│  $ lscpu                                                    │
│  CPU(s):                32                                  │
│  Thread(s) per core:    2                                   │
│  Core(s) per socket:    8                                   │
│  Socket(s):             2                                   │
│                                                             │
│  问题：                                                      │
│    - CPU利用率只有50%                                        │
│    - 内存带宽利用率只有30%                                   │
│    - 数据库查询延迟反而比16核服务器更高！                    │
│                                                             │
│  原因：不了解NUMA架构，进程和内存分布不合理                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**为什么会出现这种情况？**

现代多核服务器采用NUMA架构，CPU和内存被划分为多个节点。如果进程在一个节点上运行，却频繁访问另一个节点的内存，就会产生**跨节点内存访问**，延迟比本地访问高1.5-2倍！

### 1.2 传统SMP架构的瓶颈

在早期的多CPU服务器中，使用的是SMP（对称多处理）架构：

```
┌─────────────────────────────────────────────────────────────┐
│                    SMP架构示意图                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                    ┌─────────────┐                          │
│                    │   系统总线   │                          │
│                    └──────┬──────┘                          │
│                           │                                  │
│        ┌──────────────────┼──────────────────┐              │
│        │                  │                  │              │
│        ▼                  ▼                  ▼              │
│  ┌──────────┐       ┌──────────┐       ┌──────────┐        │
│  │   CPU0   │       │   CPU1   │       │   CPU2   │        │
│  └──────────┘       └──────────┘       └──────────┘        │
│        │                  │                  │              │
│        └──────────────────┼──────────────────┘              │
│                           │                                  │
│                    ┌──────┴──────┐                          │
│                    │   共享内存   │                          │
│                    └─────────────┘                          │
│                                                             │
│  问题分析：                                                  │
│    ┌─────────────────────────────────────────────────┐      │
│    │ 1. 所有CPU共享一条总线                           │      │
│    │ 2. CPU数量增加时总线成为瓶颈                     │      │
│    │ 3. 内存访问延迟相同但带宽受限                    │      │
│    │ 4. 扩展性差，通常不超过8核                       │      │
│    └─────────────────────────────────────────────────┘      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**SMP架构的核心问题**：

| 问题 | 说明 | 影响 |
|------|------|------|
| 总线瓶颈 | 所有CPU共享一条总线 | CPU越多，竞争越激烈 |
| 带宽受限 | 总线带宽固定 | 无法随CPU数量扩展 |
| 扩展性差 | 最多支持8核左右 | 无法满足现代需求 |
| 延迟相同 | 所有CPU访问延迟一样 | 无法优化热点数据 |

### 1.3 NUMA架构的解决方案

NUMA（Non-Uniform Memory Access，非统一内存访问）架构通过将CPU和内存分组来解决SMP的瓶颈：

```
┌─────────────────────────────────────────────────────────────┐
│                    NUMA架构示意图                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│              ┌───────────────────────────────┐              │
│              │     高速互连总线(QPI/IF)       │              │
│              └───────────────┬───────────────┘              │
│                              │                               │
│        ┌─────────────────────┴─────────────────────┐        │
│        │                                           │        │
│        ▼                                           ▼        │
│  ┌─────────────────┐                     ┌─────────────────┐│
│  │    Node 0       │                     │    Node 1       ││
│  ├─────────────────┤                     ├─────────────────┤│
│  │ ┌─────┐┌─────┐ │                     │ ┌─────┐┌─────┐ ││
│  │ │CPU0 ││CPU1 │ │                     │ │CPU2 ││CPU3 │ ││
│  │ └─────┘└─────┘ │                     │ └─────┘└─────┘ ││
│  │ ┌─────────────┐│                     │ ┌─────────────┐││
│  │ │  本地内存   ││                     │ │  本地内存   │││
│  │ │   64GB      ││                     │ │   64GB      │││
│  │ └─────────────┘│                     │ └─────────────┘││
│  └─────────────────┘                     └─────────────────┘│
│                                                             │
│  优势：                                                      │
│    ┌─────────────────────────────────────────────────┐      │
│    │ 1. 每个节点有独立的CPU和本地内存                  │      │
│    │ 2. 本地内存访问延迟低                            │      │
│    │ 3. 远程内存访问延迟高（1.5-2倍）                 │      │
│    │ 4. 扩展性好，可支持数百核心                      │      │
│    └─────────────────────────────────────────────────┘      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.4 NUMA的核心价值

```
┌─────────────────────────────────────────────────────────────┐
│                    NUMA核心价值                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 解决扩展性问题                                           │
│     - 每个节点独立的内存控制器                               │
│     - 总带宽随节点数量线性增长                               │
│     - 可扩展到数百甚至上千核心                               │
│                                                             │
│  2. 提供优化空间                                             │
│     - 本地访问快，可针对性优化                               │
│     - 热点数据可放在本地内存                                 │
│     - 进程可绑定到数据所在节点                               │
│                                                             │
│  3. 降低成本                                                 │
│     - 无需昂贵的共享总线                                     │
│     - 使用标准互连技术                                       │
│     - 模块化设计便于维护                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、NUMA架构核心概念

## 四、Linux内核NUMA数据结构

### 4.1 pglist_data（节点描述符）

**pglist_data是NUMA节点的核心数据结构**，定义在 **`include/linux/mmzone.h`**：

```c
/*
 * pglist_data - 描述一个NUMA节点的内存布局
 * 
 * 在NUMA机器上，每个NUMA节点都有一个pg_data_t来描述其内存布局
 * 在UMA机器上，只有一个pglist_data描述整个内存
 */
typedef struct pglist_data {
    /*
     * node_zones: 该节点的内存区域数组
     * 包含DMA、DMA32、NORMAL、HIGHMEM、MOVABLE等区域
     */
    struct zone node_zones[MAX_NR_ZONES];

    /*
     * node_zonelists: 内存分配时使用的区域列表
     * 包含本节点和其他节点的区域引用，用于fallback分配
     */
    struct zonelist node_zonelists[MAX_ZONELISTS];

    int nr_zones;              /* 该节点已填充的区域数量 */

    unsigned long node_start_pfn;      /* 节点起始页帧号 */
    unsigned long node_present_pages;  /* 节点物理页总数 */
    unsigned long node_spanned_pages;  /* 节点跨度页数（含空洞） */

    int node_id;               /* 节点ID */

    wait_queue_head_t kswapd_wait;     /* kswapd等待队列 */
    struct task_struct *kswapd;        /* 该节点的kswapd内核线程 */
    int kswapd_order;                  /* kswapd回收阶数 */
```

### 5.3 策略获取函数

```c
/*
 * get_task_policy - 获取任务的内存策略
 * 
 * 策略查找顺序：
 * 1. 任务自身的策略
 * 2. 当前节点的preferred策略
 * 3. 系统默认策略
 */
struct mempolicy *get_task_policy(struct task_struct *p)
{
    struct mempolicy *pol = p->mempolicy;
    int node;

    /* 如果任务有策略，直接返回 */
    if (pol)
        return pol;

    /* 尝试使用当前节点的preferred策略 */
    node = numa_node_id();
    if (node != NUMA_NO_NODE) {
        pol = &preferred_node_policy[node];
        if (pol->mode)
            return pol;
    }

    /* 返回默认策略 */
    return &default_policy;
}
```

### 7.2 NUMA调度核心函数

定义在 **`kernel/sched/fair.c`**：

```c
/*
 * task_numa_find_cpu - 为任务找到最佳CPU
 * 
 * 考虑因素：
 * 1. CPU负载
 * 2. 内存亲和性
 * 3. 迁移代价
 */
static int task_numa_find_cpu(struct task_struct *p, int nid)
{
    struct numa_stats *ns;
    int cpu, best_cpu = -1;
    unsigned long best_score = 0;

    /* 遍历目标节点的所有CPU */
    for_each_cpu(cpu, cpumask_of_node(nid)) {
        unsigned long score;

        /* 计算该CPU的分数 */
        score = task_numa_score(p, cpu);

        /* 选择分数最高的CPU */
        if (score > best_score) {
            best_score = score;
            best_cpu = cpu;
        }
    }

    return best_cpu;
}

/*
 * task_numa_migrate - 将任务迁移到最优节点
 */
static void task_numa_migrate(struct task_struct *p)
{
    int node, best_node = -1;
    unsigned long best_score = 0;

    /* 遍历所有节点，计算分数 */
    for_each_online_node(node) {
        unsigned long score;

        score = task_numa_compare(p, node);
        if (score > best_score) {
            best_score = score;
            best_node = node;
        }
    }

    /* 迁移到最优节点 */
    if (best_node != -1 && best_node != numa_node_id())
        migrate_task_to(p, best_node);
}
```

### 8.3 使用numactl绑定进程

```bash
# 在Node 0上分配内存并运行
numactl --cpunodebind=0 --membind=0 ./my_program

# 在指定CPU上运行，从Node 0分配内存
numactl --physcpubind=0-3 --membind=0 ./my_program

# 交错分配内存（适合大内存应用）
numactl --interleave=all ./my_program

# 首选Node 0，但允许fallback
numactl --preferred=0 ./my_program
```

### 8.4 numactl参数详解

| 参数 | 说明 | 示例 |
|------|------|------|
| --membind=nodes | 只从指定节点分配内存 | --membind=0 |
| --cpunodebind=nodes | 在指定节点的CPU上运行 | --cpunodebind=0,1 |
| --physcpubind=cpus | 在指定物理CPU上运行 | --physcpubind=0-3 |
| --interleave=nodes | 在节点间交错分配 | --interleave=all |
| --preferred=node | 首选节点 | --preferred=0 |

### 8.5 编程接口示例

```c
#include <numaif.h>
#include <numa.h>
#include <sys/mman.h>

void numa_example(void)
{
    void *mem;
    unsigned long nodemask;
    
    /* 方法1：使用set_mempolicy */
    nodemask = 0x1;  /* 只使用Node 0 */
    set_mempolicy(MPOL_BIND, &nodemask, 2);
    mem = malloc(1024 * 1024);  /* 从Node 0分配 */
    
    /* 方法2：使用mbind绑定特定内存区域 */
    mem = mmap(NULL, 1024 * 1024, PROT_READ | PROT_WRITE,
               MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
    nodemask = 0x2;  /* 只使用Node 1 */
    mbind(mem, 1024 * 1024, MPOL_BIND, &nodemask, 2, 0);
    
    /* 方法3：交错分配 */
    nodemask = 0x3;  /* 使用Node 0和Node 1 */
    set_mempolicy(MPOL_INTERLEAVE, &nodemask, 2);
    mem = malloc(1024 * 1024 * 100);  /* 大内存交错分配 */
}
```

### 8.6 内核参数配置

```bash

# 4 = 允许将页面换出
```

## 总结

NUMA架构是现代多核服务器的核心设计，理解它对于系统性能优化至关重要：

1. **核心概念**：Node、本地访问、远程访问、节点距离
2. **内核实现**：pglist_data节点描述符、mempolicy内存策略
3. **优化机制**：NUMA平衡、NUMA感知调度
4. **实践应用**：numactl工具、编程接口、应用优化

**关键要点**：
- 本地访问比远程访问快1.5-2倍
- 合理的NUMA策略可以提升20-30%性能
- 不同应用场景需要不同的NUMA策略
- Linux内核提供了完整的NUMA支持机制

## 一、引言：为什么需要TC

### 1.1 传统网络流量管理的问题

在Linux系统中，默认情况下网络流量是"尽力而为"（Best-Effort）的，即所有流量平等竞争带宽资源。这种简单的方式在很多场景下会带来问题：

```
┌─────────────────────────────────────────────────────────────┐
│                传统网络流量管理的问题                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  问题1：关键流量被挤占                                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  场景：公司网络                                          │   │
│  │  • 员工下载大文件占用全部带宽                            │   │
│  │  • VoIP电话、视频会议卡顿                                │   │
│  │  • 重要业务访问缓慢                                      │   │
│  │                                                          │   │
│  │  结果：关键业务无法正常工作                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  问题2：无法限制特定应用                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  场景：P2P下载、视频流                                   │   │
│  │  • P2P软件占用所有上传带宽                              │   │
│  │  • 视频流占用大量下载带宽                                │   │
│  │  • 其他应用无法正常使用网络                              │   │
│  │                                                          │   │
│  │  结果：网络拥塞，用户体验差                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  问题3：无法保证服务质量（QoS）                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  场景：实时应用                                          │   │
│  │  • VoIP需要低延迟和低抖动                                │   │
│  │  • 视频会议需要稳定带宽                                  │   │
│  │  • 游戏需要低延迟                                        │   │
│  │                                                          │   │
│  │  但默认情况下无法提供这些保证                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  问题4：ISP带宽限制                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  场景：家庭网络                                          │   │
│  │  • ISP限制上传速度为1Mbps                                │   │
│  │  • 但Linux默认会尽可能快地发送                           │   │
│  │  • 导致ISP端队列堆积，延迟增加                           │   │
│  │                                                          │   │
│  │  结果：网络延迟大，丢包多                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  问题5：多租户环境公平性                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  场景：云服务器、VPS                                     │   │
│  │  • 多个租户共享带宽                                      │   │
│  │  • 某个租户可能占用过多资源                              │   │
│  │  • 其他租户受影响                                        │   │
│  │                                                          │   │
│  │  结果：资源分配不公平                                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```
│  │                                                          │   │

│                                                             │
│  解决方案4：公平队列                                          │

│  │  • 多个流公平竞争带宽                                    │   │

│  │  • 控制队列长度，降低延迟                                │   │
│  │                                                          │   │
│  │  效果：减少网络拥塞，提高吞吐量                          │   │

│                                                             │
```

### 1.3 TC的实际应用场景

| 应用场景 | TC的作用 | 效果 |
|---------|---------|------|
| 家庭路由器 | 限制P2P上传，保障游戏延迟 | 游戏不卡，下载继续 |

**关键点：TC在驱动发送之前介入**

TC位于网络协议栈的发送路径上，在数据包交给网卡驱动之前，TC有机会：
- 决定包的发送顺序
- 延迟包的发送
- 丢弃包
- 修改包的优先级

### 2.2 默认的qdisc行为

Linux默认为每个网卡配置一个pfifo_fast qdisc：

```
┌─────────────────────────────────────────────────────────────┐
│                默认pfifo_fast qdisc                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  配置：                                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  # tc qdisc show dev eth0                             │   │
│  │  qdisc pfifo_fast 0: root refcnt 2 bands 3            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  工作原理：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  三个优先级队列（band）：                               │   │
│  │                                                         │   │
│  │  Band 0（高优先级）──→ TOS低延迟、交互流量             │   │
│  │  Band 1（中优先级）──→ 普通流量                        │   │
│  │  Band 2（低优先级）──→ TOS高吞吐、批量流量             │   │
│  │                                                         │   │
│  │  出队顺序：Band 0 → Band 1 → Band 2                    │   │
│  │  （高优先级队列空了才出下一个队列）                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  局限性：                                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 无法限速                                             │   │
│  │  • 无法保证带宽                                         │   │
│  │  • 仅基于TOS字段分类，粒度粗                            │   │
│  │  • 无法区分不同应用或用户                               │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

QoS是一套用于保证网络服务质量的技术体系：

```
│                                                             │

│  ┌─────────────────────────────────────────────────────┐   │
│  │  定义：延迟的变化程度                                   │   │
│  │  单位：毫秒（ms）                                       │   │

│                                                             │
│  4. 丢包率（Packet Loss）                                    │

│  │  定义：丢失的数据包占总发送包的比例                     │   │
│  │  单位：百分比（%）                                      │   │
│  │  示例：VoIP要求丢包率 < 1%                              │   │

│                                                             │
│  不同应用的QoS需求：                                         │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  VoIP：低延迟、低抖动、低丢包率                         │   │
│  │  视频流：稳定带宽、中等延迟                             │   │

│  │  文件下载：高带宽、延迟不敏感                           │   │

│                                                             │
```

## 三、TC概述

### 3.1 什么是TC

**TC（Traffic Control）** 是Linux内核提供的流量控制框架，用于实现QoS、限速、流量整形等功能。

简单来说，TC就是**网络流量的"交通警察"**：
- 决定哪些包先走，哪些包后走（调度）
- 决定哪些包可以走，哪些包不能走（限速）
- 决定哪些包走快车道，哪些包走慢车道（分类）

```
│  ┌─────────────────────────────────────────────────────┐   │
```

```
│  │  ret = q->enqueue(skb, q, &to_free)                   │   │
│  │                                                         │   │
│  │  对于有类qdisc（如HTB）：                               │   │
│  │    2.1 调用filter进行分类                               │   │
│  │        cl = htb_classify(skb, q, &ret)                │   │
│  │                                                         │   │
│  │    2.2 根据分类结果选择类                               │   │
│  │        if (cl == HTB_DIRECT) → 直接队列                │   │
│  │        else → cl->leaf.q                               │   │
│  │                                                         │   │
│  │    2.3 将包加入对应类的队列                             │   │
│  │        qdisc_enqueue(skb, cl->leaf.q)                  │   │
│  │                                                         │   │
│  │    2.4 激活该类（加入调度队列）                         │   │
│  │        htb_activate(q, cl)                             │   │
│  └─────────────────────────────────────────────────────┘   │
│                           │                                 │
│                           ▼                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  步骤3：触发发送（如果队列不忙）                         │   │
│  │  netif_schedule_tx(dev)                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                           │                                 │
│                           ▼                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  步骤4：驱动发送线程调用dequeue                         │   │
│  │  while (true) {                                        │   │
│  │      skb = q->dequeue(q);                              │   │
│  │      if (!skb) break;                                  │   │
│  │      dev->netdev_ops->ndo_start_xmit(skb, dev);        │   │
│  │  }                                                      │   │
│  │                                                         │   │
│  │  对于HTB：                                              │   │
│  │    4.1 更新令牌                                         │   │
│  │    4.2 选择有令牌的类                                   │   │
│  │    4.3 从该类出队一个包                                 │   │
│  │    4.4 消耗令牌                                         │   │
│  └─────────────────────────────────────────────────────┘   │
│                           │                                 │
│                           ▼                                 │
│  ┌─────────────────────────────────────────────────────┐   │
```

## 四、TC核心数据结构

### 2.1 Qdisc结构

```c
/* include/net/sch_generic.h */

/*
 * Qdisc - 队列规则核心结构
 * 
 * 这是TC最核心的数据结构
 * 每个网络设备都有一个或多个qdisc
 */
struct Qdisc {
    /* 入队函数（必须实现） */
    int (*enqueue)(struct sk_buff *skb,
                   struct Qdisc *sch,
                   struct sk_buff **to_free);
    
    /* 出队函数（必须实现） */
    struct sk_buff *(*dequeue)(struct Qdisc *sch);
    
    /* 标志位 */
    unsigned int flags;
    #define TCQ_F_BUILTIN      1    /* 内置qdisc */
    #define TCQ_F_INGRESS      2    /* 入口qdisc */
    #define TCQ_F_CAN_BYPASS   4    /* 可以绕过 */
    #define TCQ_F_MQROOT       8    /* 多队列根 */
    #define TCQ_F_CPUSTATS     0x20 /* 使用per-CPU统计 */
    #define TCQ_F_NOLOCK       0x100 /* 无锁qdisc */
    
    /* 队列限制 */
    u32 limit;
    
    /* 操作函数表 */
    const struct Qdisc_ops *ops;
    
    /* 尺寸表 */
    struct qdisc_size_table __rcu *stab;
    
    /* 哈希节点 */
    struct hlist_node hash;
    
    /* 句柄（标识符） */
    u32 handle;    /* 格式：MAJOR:MINOR，如10:0 */
    
    /* 父句柄 */
    u32 parent;    /* TC_H_ROOT表示根qdisc */
    
    /* 关联的网络设备队列 */
    struct netdev_queue *dev_queue;
    
    /* 速率估算器 */
    struct net_rate_estimator __rcu *rate_est;
    
    /* 统计信息 */
    struct gnet_stats_basic_packed bstats;  /* 字节/包统计 */
    struct gnet_stats_queue qstats;          /* 队列统计 */
    
    /* 队列状态 */
    unsigned long state;
    
    /* 数据包队列 */
    struct qdisc_skb_head q;    /* 队列头 */
    
    /* 引用计数 */
    refcount_t refcnt;
    
    /* 私有数据 */
    long privdata[] ____cacheline_aligned;
};
```

**关键字段说明：**

1. **enqueue**: 入队函数
   - 将包加入队列
   - 返回NET_XMIT_SUCCESS或错误码
   - 可以修改包或丢弃包

2. **dequeue**: 出队函数
   - 从队列取出一个包
   - 返回skb指针或NULL
   - 实现调度逻辑

3. **handle**: 句柄
   - 唯一标识一个qdisc
   - 格式：MAJOR:MINOR（如10:0）
   - qdisc的MINOR通常为0

4. **parent**: 父句柄
   - TC_H_ROOT (0xFFFFFFFF): 根qdisc
   - TC_H_INGRESS (0xFFFFFFF1): 入口qdisc
   - 其他：父类的句柄

5. **ops**: 操作函数表
   - 指向Qdisc_ops结构
   - 包含各种操作函数

### 2.2 Qdisc_ops结构

```c
/*
 * Qdisc_ops - qdisc操作函数表
 * 
 * 每种qdisc类型都要实现这个结构
 */
struct Qdisc_ops {
    /* 下一个qdisc操作（链表） */
    struct Qdisc_ops    *next;
    
    /* qdisc类型名称 */
    char                id[IFNAMSIZ];
    
    /* 私有数据大小 */
    int                 priv_size;
    
    /* 入队函数 */
    int                 (*enqueue)(struct sk_buff *skb,
                                    struct Qdisc *sch,
                                    struct sk_buff **to_free);
    
    /* 出队函数 */
    struct sk_buff *    (*dequeue)(struct Qdisc *sch);
    
    /* 窥视函数（查看队首包） */
    struct sk_buff *    (*peek)(struct Qdisc *sch);
    
    /* 初始化函数 */
    int                 (*init)(struct Qdisc *sch,
                                struct nlattr *opt,
                                struct netlink_ext_ack *extack);
    
    /* 重置函数 */
    void                (*reset)(struct Qdisc *sch);
    
    /* 销毁函数 */
    void                (*destroy)(struct Qdisc *sch);
    
    /* 改变函数（修改参数） */
    int                 (*change)(struct Qdisc *sch,
                                  struct nlattr *opt,
                                  struct netlink_ext_ack *extack);
    
    /* 转储函数（导出统计信息） */
    int                 (*dump)(struct Qdisc *sch,
                                struct sk_buff *skb);
    
    /* 模块指针 */
    struct module       *owner;
    
    /* 类操作（如果有） */
    const struct Qdisc_class_ops *cl_ops;
};
```

### 2.3 htb_class结构

```c
/* net/sched/sch_htb.c */

/*
 * htb_class - HTB类结构
 * 
 * HTB的核心数据结构
 * 每个类都有自己的速率限制和优先级
 */
struct htb_class {
    /* 通用类信息 */
    struct Qdisc_class_common common;
    
    /* 保证速率（rate） */
    struct psched_ratecfg rate;
    
    /* 上限速率（ceil） */
    struct psched_ratecfg ceil;
    
    /* 令牌桶深度 */
    s64 buffer, cbuffer;    /* rate和ceil的桶深度 */
    s64 mbuffer;            /* 最大等待时间 */
    
    /* 优先级和量子 */
    u32 prio;               /* 优先级 */
    int quantum;            /* DRR的量子值 */
    
    /* 过滤器链 */
    struct tcf_proto __rcu *filter_list;
    struct tcf_block *block;
    int filter_cnt;
    
    /* 层次级别 */
    int level;              /* 0表示叶子类 */
    unsigned int children;  /* 子类数量 */
    struct htb_class *parent; /* 父类指针 */
    
    /* 速率估算器 */
    struct net_rate_estimator __rcu *rate_est;
    
    /* 统计信息 */
    struct gnet_stats_basic_packed bstats;
    struct tc_htb_xstats xstats;
    
    /* 令牌桶状态 */
    s64 tokens, ctokens;    /* 当前令牌数 */
    s64 t_c;                /* 检查点时间 */
    
    /* 叶子类或内部类的联合体 */
    union {
        struct htb_class_leaf {
            int deficit[TC_HTB_MAXDEPTH];
            struct Qdisc *q;    /* 叶子类的qdisc */
            struct netdev_queue *offload_queue;
        } leaf;
        struct htb_class_inner {
            struct htb_prio clprio[TC_HTB_NUMPRIO];
        } inner;
    };
    
    /* 活动状态 */
    int prio_activity;      /* 活动优先级位图 */
    enum htb_cmode cmode;   /* 当前模式 */
    
    /* 红黑树节点 */
    struct rb_node pq_node;
    struct rb_node node[TC_HTB_NUMPRIO];
    
    /* 统计 */
    unsigned int drops;
    unsigned int overlimits;
};
```

**关键字段说明：**

1. **rate**: 保证速率
   - 类保证获得的带宽
   - 即使拥塞也能保证

2. **ceil**: 上限速率
   - 类的最大带宽
   - 可以借用父类的带宽

3. **level**: 层次级别
   - 0: 叶子类（包含实际的qdisc）
   - >0: 内部类（包含子类）

4. **tokens/ctokens**: 令牌数
   - tokens: rate令牌桶
   - ctokens: ceil令牌桶

5. **cmode**: 当前模式
   - HTB_CANT_SEND: 不能发送
   - HTB_MAY_BORROW: 可以借用
   - HTB_CAN_SEND: 可以发送

### 3.2 pfifo_fast详解

```c
/*
 * pfifo_fast - 默认qdisc
 * 
 * 3个优先级队列：
 *   band 0: 高优先级（交互式流量）
 *   band 1: 中优先级（最佳努力流量）
 *   band 2: 低优先级（批量流量）
 */
struct pfifo_fast_priv {
    struct sk_buff_head q[3];  /* 3个队列 */
};

/* 优先级映射表 */
static const u8 prio2band[TC_PRIO_MAX + 1] = {
    1,      /* 0: Best effort → band 1 */
    2,      /* 1: Filler → band 2 */
    2,      /* 2: Bulk → band 2 */
    2,      /* 3: */
    1,      /* 4: Interactive bulk → band 1 */
    2,      /* 5: */
    0,      /* 6: Interactive → band 0 */
    0       /* 7: Control → band 0 */
};
```

## 八、HTB调度器详解

### 6.1 HTB原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    HTB工作原理                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  核心概念：                                                     │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ 1. 令牌桶算法                                         │       │
│  │    • 每个类有两个令牌桶：rate和ceil                   │       │
│  │    • rate: 保证速率，令牌以rate速率生成               │       │
│  │    • ceil: 上限速率，令牌以ceil速率生成               │       │
│  │    • 发送包需要消耗令牌                               │       │
│  │    • 令牌不足则等待                                   │       │
│  └─────────────────────────────────────────────────────┘       │
│                                                                 │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ 2. 层次化结构                                         │       │
│  │    • 类形成树状结构                                   │       │
│  │    • 每个类有level（层次级别）                        │       │
│  │    • 叶子类level=0，包含实际的qdisc                   │       │
│  │    • 内部类level>0，包含子类                          │       │
│  │    • level越小优先级越高                              │       │
│  └─────────────────────────────────────────────────────┘       │
│                                                                 │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ 3. 三种模式                                           │       │
│  │    • HTB_CANT_SEND: 不能发送                         │       │
│  │      - rate和ceil令牌都不足                          │       │
│  │    • HTB_MAY_BORROW: 可以借用                        │       │
│  │      - rate令牌不足，ceil令牌充足                     │       │
│  │      - 可以向父类借用带宽                             │       │
│  │    • HTB_CAN_SEND: 可以发送                          │       │
│  │      - rate令牌充足                                  │       │
│  └─────────────────────────────────────────────────────┘       │
│                                                                 │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ 4. 借用机制                                           │       │
│  │    • 当类处于MAY_BORROW模式时                        │       │
│  │    • 可以向父类借用带宽                               │       │
│  │    • 借用受ceil限制                                   │       │
│  │    • 优先级高的类优先借用                             │       │
│  └─────────────────────────────────────────────────────┘       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 HTB令牌桶算法

```
┌─────────────────────────────────────────────────────────────────┐
│                    HTB令牌桶算法                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  令牌桶参数：                                                   │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ • rate: 保证速率（字节/秒）                           │       │
│  │ • ceil: 上限速率（字节/秒）                           │       │
│  │ • buffer: rate桶深度（字节）                          │       │
│  │ • cbuffer: ceil桶深度（字节）                         │       │
│  │ • tokens: 当前rate令牌数                              │       │
│  │ • ctokens: 当前ceil令牌数                             │       │
│  └─────────────────────────────────────────────────────┘       │
│                                                                 │
│  令牌更新（时间从t_c到now）：                                   │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ diff = now - t_c;                                     │       │
│  │                                                       │       │
│  │ /* 更新rate令牌 */                                    │       │
│  │ tokens += diff * rate / NSEC_PER_SEC;                 │       │
│  │ if (tokens > buffer)                                  │       │
│  │     tokens = buffer;  /* 不超过桶深度 */              │       │
│  │                                                       │       │
│  │ /* 更新ceil令牌 */                                    │       │
│  │ ctokens += diff * ceil / NSEC_PER_SEC;                │       │
│  │ if (ctokens > cbuffer)                                │       │
│  │     ctokens = cbuffer;                                │       │
│  │                                                       │       │
│  │ t_c = now;  /* 更新检查点时间 */                      │       │
│  └─────────────────────────────────────────────────────┘       │
│                                                                 │
│  发送包（消耗令牌）：                                           │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ bytes = skb->len;                                     │       │
│  │                                                       │       │
│  │ /* 消耗rate令牌 */                                    │       │
│  │ tokens -= bytes;                                      │       │
│  │                                                       │       │
│  │ /* 消耗ceil令牌 */                                    │       │
│  │ ctokens -= bytes;                                     │       │
│  └─────────────────────────────────────────────────────┘       │
│                                                                 │
│  模式判断：                                                     │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ if (tokens >= 0)                                      │       │
│  │     mode = HTB_CAN_SEND;                              │       │
│  │ else if (ctokens >= 0)                                │       │
│  │     mode = HTB_MAY_BORROW;                            │       │
│  │ else                                                  │       │
│  │     mode = HTB_CANT_SEND;                             │       │
```

## 九、htb_enqueue()流程分析

### 7.1 htb_enqueue()函数

```c
/* net/sched/sch_htb.c */

static int htb_enqueue(struct sk_buff *skb, struct Qdisc *sch,
                       struct sk_buff **to_free)
{
    int ret;
    unsigned int len = qdisc_pkt_len(skb);
    struct htb_sched *q = qdisc_priv(sch);
    struct htb_class *cl = htb_classify(skb, sch, &ret);
    
    /* 情况1: 直接发送队列 */
    if (cl == HTB_DIRECT) {
        /* 加入直接发送队列 */
        if (q->direct_queue.qlen < q->direct_qlen) {
            __qdisc_enqueue_tail(skb, &q->direct_queue);
            q->direct_pkts++;
        } else {
            return qdisc_drop(skb, sch, to_free);
        }
    }
    /* 情况2: 分类失败（如TC_ACT_SHOT） */
    else if (!cl) {
        if (ret & __NET_XMIT_BYPASS)
            qdisc_qstats_drop(sch);
        __qdisc_drop(skb, to_free);
        return ret;
    }
    /* 情况3: 正常分类到叶子类 */
    else if ((ret = qdisc_enqueue(skb, cl->leaf.q,
                                  to_free)) != NET_XMIT_SUCCESS) {
        /* 叶子qdisc入队失败 */
        if (net_xmit_drop_count(ret)) {
            qdisc_qstats_drop(sch);
            cl->drops++;
        }
        return ret;
    } else {
        /* 成功入队，激活类 */
        htb_activate(q, cl);
    }
    
    /* 更新统计 */
    sch->qstats.backlog += len;
    sch->q.qlen++;
    return NET_XMIT_SUCCESS;
}
```

## 十、htb_dequeue()流程分析

### 8.1 htb_dequeue()函数

```c
static struct sk_buff *htb_dequeue(struct Qdisc *sch)
{
    struct sk_buff *skb;
    struct htb_sched *q = qdisc_priv(sch);
    int level;
    s64 next_event;
    unsigned long start_at;
    
    /* 1. 尝试从直接发送队列取包 */
    skb = __qdisc_dequeue_head(&q->direct_queue);
    if (skb != NULL) {
ok:
        qdisc_bstats_update(sch, skb);
        qdisc_qstats_backlog_dec(sch, skb);
        sch->q.qlen--;
        return skb;
    }
    
    /* 2. 检查队列是否为空 */
    if (!sch->q.qlen)
        goto fin;
    
    /* 3. 获取当前时间 */
    q->now = ktime_get_ns();
    start_at = jiffies;
    
    /* 4. 初始化下一个事件时间 */
    next_event = q->now + 5LLU * NSEC_PER_SEC;
    
    /* 5. 从level 0开始，逐层尝试出队 */
    for (level = 0; level < TC_HTB_MAXDEPTH; level++) {
        int m;
        s64 event = q->near_ev_cache[level];
        
        /* 5.1 处理事件（更新令牌等） */
        if (q->now >= event) {
            event = htb_do_events(q, level, start_at);
            if (!event)
                event = q->now + NSEC_PER_SEC;
            q->near_ev_cache[level] = event;
        }
        
        /* 5.2 更新下一个事件时间 */
        if (next_event > event)
            next_event = event;
        
        /* 5.3 遍历该层的所有优先级 */
        m = ~q->row_mask[level];
        while (m != (int)(-1)) {
            int prio = ffz(m);
            
            m |= 1 << prio;
            
            /* 5.4 从该优先级出队 */
            skb = htb_dequeue_tree(q, prio, level);
            if (likely(skb != NULL))
                goto ok;
        }
    }
    
    /* 6. 设置定时器，等待下一个事件 */
    if (likely(next_event > q->now))
        qdisc_watchdog_schedule_ns(&q->watchdog, next_event);
    else
        schedule_work(&q->work);
    
fin:
    return skb;
}
```

### 9.2 示例1：使用HTB限制带宽为1Mbps（★重点示例）

```bash
#!/bin/bash
# htb_1mbps.sh - 使用HTB限制带宽为1Mbps

# 网络接口
IFACE="eth0"

# 清除现有配置
tc qdisc del dev $IFACE root 2>/dev/null

# 步骤1: 添加根qdisc（HTB）
# handle 10: 表示qdisc的句柄为10:
# default 20 表示未分类的包发送到类10:20
tc qdisc add dev $IFACE root handle 10: htb default 20

# 步骤2: 添加根类
# parent 10: 表示父句柄为10:（即根qdisc）
# classid 10:1 表示类的ID为10:1
# rate 10mbit 保证速率10Mbps
# ceil 10mbit 上限速率10Mbps
tc class add dev $IFACE parent 10: classid 10:1 htb rate 10mbit ceil 10mbit

# 步骤3: 添加叶子类（限制为1Mbps）
# parent 10:1 表示父类为10:1
# classid 10:10 表示类的ID为10:10
# rate 1mbit 保证速率1Mbps
# ceil 1mbit 上限速率1Mbps（不能超过1Mbps）
# prio 1 优先级为1
tc class add dev $IFACE parent 10:1 classid 10:10 htb rate 1mbit ceil 1mbit prio 1

# 步骤4: 添加默认类
# 未分类的包发送到这里
tc class add dev $IFACE parent 10:1 classid 10:20 htb rate 9mbit ceil 9mbit prio 2

# 步骤5: 为叶子类添加qdisc
# 类10:10使用sfq（随机公平队列）
tc qdisc add dev $IFACE parent 10:10 handle 10:10 sfq perturb 10

# 类10:20使用sfq
tc qdisc add dev $IFACE parent 10:20 handle 10:20 sfq perturb 10

# 步骤6: 添加过滤器（可选）
# 将端口80的流量分类到类10:10
tc filter add dev $IFACE parent 10: protocol ip prio 1 u32 match ip dport 80 0xffff flowid 10:10

# 显示配置
echo "=== Qdisc配置 ==="
tc qdisc show dev $IFACE

echo -e "\n=== Class配置 ==="
tc class show dev $IFACE

echo -e "\n=== Filter配置 ==="
tc filter show dev $IFACE

echo -e "\n配置完成！端口80的流量被限制为1Mbps"
```

**配置说明：**

1. **根qdisc (10:)**
   - 类型：HTB
   - 默认类：10:20

2. **根类 (10:1)**
   - 总带宽：10Mbps
   - 作用：限制所有流量的总和

3. **叶子类 (10:10)**
   - 保证速率：1Mbps
   - 上限速率：1Mbps
   - 优先级：1（高）
   - 用途：Web流量（端口80）

4. **默认类 (10:20)**
   - 保证速率：9Mbps
   - 上限速率：9Mbps
   - 优先级：2（低）
   - 用途：其他流量

## 总结

### 核心要点回顾

1. **TC框架**
   - Linux内核流量控制的核心框架
   - 通过qdisc、class、filter三大组件实现
   - 支持QoS、限速、整形等功能

2. **qdisc（队列规则）**
   - 控制包的发送顺序和时机
   - 分为无类qdisc和有类qdisc
   - HTB是最常用的有类qdisc

3. **class（类）**
   - 实现流量的分类和层次化管理
   - 每个类可以设置保证速率和上限速率
   - 支持带宽借用机制

4. **filter（过滤器）**
   - 决定包属于哪个类
   - 支持u32、fw等多种匹配规则
   - 可以根据端口、IP、mark等分类

5. **HTB调度器**
   - 基于令牌桶算法
   - 支持层次化结构和带宽借用
   - htb_enqueue()和htb_dequeue()是核心函数

6. **tc命令示例**
   - 提供了完整的HTB配置示例
   - 实现了限制带宽为1Mbps的目标
   - 包含监控和调试方法

### 学习建议

1. **从简单到复杂**
   - 先理解TC框架原理
   - 再学习各种qdisc类型
   - 最后掌握HTB等复杂调度器

2. **理论结合实践**
   - 使用tc命令配置和测试
   - 观察统计信息
   - 调整参数优化效果

3. **关注关键点**
   - qdisc的enqueue和dequeue函数
   - HTB的令牌桶算法
   - 过滤器的匹配规则

4. **善用调试工具**
   - tc命令的各种选项
   - 统计信息的分析
   - 性能测试工具

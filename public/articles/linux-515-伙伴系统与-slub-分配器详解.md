# Linux 5.15 伙伴系统与 SLUB 分配器详解

## 一、从一个问题开始

### 1.1 假设你是内存管理员

想象你管理一个有 1024 个房间的酒店：

```
场景：酒店房间管理
+===========================================================================+
|                                                                           |
|   问题 1：大客户来了                                                      |
|   +-------------------+                                                    |
|   | 旅行团要 64 个连续房间 |  需要房间号连续                            |
|   | 公司要 128 个连续房间 |  方便管理                                    |
|   +-------------------+                                                    |
|                                                                           |
|   问题 2：散客来了                                                        |
|   +-------------------+                                                    |
|   | 单人要 1 个房间     |  频繁入住退房                                |
|   | 情侣要 2 个房间     |  需要快速响应                                |
|   +-------------------+                                                    |
|                                                                           |
|   问题 3：房间碎片化                                                      |
|   +-------------------+                                                    |
|   | 房间 1-10：占用     |                                                |
|   | 房间 11：空闲       |  ← 散落的空闲房间                            |
|   | 房间 12-20：占用    |                                                |
|   | 房间 21：空闲       |  ← 无法满足大客户需求                        |
|   | 房间 22-30：占用    |                                                |
|   +-------------------+                                                    |
|                                                                           |
+===========================================================================+
```

Linux 内核面临同样的问题：
- **大块内存需求**：进程需要连续的物理页面
- **小块内存需求**：内核对象频繁分配释放
- **碎片化问题**：内存变得零散，无法满足大块需求

### 1.2 Linux 的解决方案

Linux 用两层分配器解决这个问题：

```
两层分配器架构：
+===========================================================================+
|                                                                           |
|   第一层：伙伴系统（管理大块连续内存）                                    |
|   +-------------------+                                                    |
|   | 类似：酒店楼层管理 |  按 2 的幂次划分楼层                          |
|   | 最小单位：1 页     |  4KB（一个房间）                              |
|   | 最大单位：1024 页  |  4MB（整层楼）                                |
|   +-------------------+                                                    |
|                                                                           |
|   第二层：SLUB 分配器（管理小块内存）                                     |
|   +-------------------+                                                    |
|   | 类似：房间内床位   |  把房间分成多个床位                          |
|   | 最小单位：字节     |  可以分配任意大小                            |
|   | 高效复用           |  床位可以快速重新分配                        |
|   +-------------------+                                                    |
|                                                                           |
+===========================================================================+
```

## 二、伙伴系统：如何实现大块内存管理

### 2.1 核心思想：二分法

伙伴系统的核心思想非常简单：**把内存按 2 的幂次方划分**。

```
为什么是 2 的幂次方？
+===========================================================================+
|                                                                           |
|   假设有 1024 页内存：                                                    |
|                                                                           |
|   初始状态：                                                              |
|   +-------------------------------------------------------------------+   |
|   |                    1024 页（order 10）                           |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   请求 256 页：                                                           |
|   +-------------------------------------------------------------------+   |
|   |  256 页  |  256 页  |  256 页  |  256 页  |                       |   |
|   +-------------------------------------------------------------------+   |
|   | order 8  | order 8  | order 8  | order 8  |                       |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   关键：每次分裂都是一分为二，合并时二合为一                              |
|                                                                           |
+===========================================================================+
```

### 2.2 什么是"伙伴"？

**伙伴的定义**：两个大小相同、地址连续的内存块，且起始地址满足特定对齐条件。

```
伙伴关系示例：
+===========================================================================+
|                                                                           |
|   假设内存从地址 0 开始，每块 4 页（order 2）                             |
|                                                                           |
|   地址范围      块编号    伙伴是谁？                                      |
|   +-------------------+                                                    |
|   | 0-3 页    块 0    |  伙伴是块 1（地址 4 XOR 4 = 0）                |
|   | 4-7 页    块 1    |  伙伴是块 0（地址 0 XOR 4 = 4）                |
|   | 8-11 页   块 2    |  伙伴是块 3（地址 12 XOR 4 = 8）               |
|   | 12-15 页  块 3    |  伙伴是块 2（地址 8 XOR 4 = 12）               |
|   +-------------------+                                                    |
|                                                                           |
|   计算公式：伙伴地址 = 当前地址 XOR (1 << order)                          |
|                                                                           |
|   例如：                                                                  |
|   - 块 0 地址 = 0，order = 2                                              |
|   - 伙伴地址 = 0 XOR 4 = 4（块 1 的起始地址）                             |
|                                                                           |
+===========================================================================+
```

### 2.3 数据结构：如何组织空闲内存

伙伴系统用**链表数组**来管理不同大小的空闲块：

```c
/*
 * free_area 结构 - 管理每个阶的空闲内存块
 * 
 * 想象成：每个楼层有一个登记簿，记录空闲房间
 */
struct free_area {
    struct list_head free_list[MIGRATE_TYPES];  /* 空闲块链表 */
    unsigned long nr_free;                       /* 空闲块数量 */
};

/*
 * zone 结构 - 内存区域
 * 
 * 想象成：酒店的一栋楼，包含多个楼层
 */
struct zone {
    /* 核心：11 个 free_area，分别管理 1, 2, 4, 8, ... 1024 页的空闲块 */
    struct free_area free_area[MAX_ORDER];  /* MAX_ORDER = 11 */
    
    /* 其他信息 */
    unsigned long zone_start_pfn;    /* 起始页帧号 */
    const char *name;                /* 区域名称 */
};
```

```
free_area 数组的工作方式：
+===========================================================================+
|                                                                           |
|   free_area[0]：管理所有 1 页的空闲块（order 0）                         |
|   +-------------------------------------------------------------------+   |
|   | free_list → [页A] → [页B] → [页C] → ...                          |   |
|   | nr_free = 5                                                       |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   free_area[1]：管理所有 2 页的空闲块（order 1）                         |
|   +-------------------------------------------------------------------+   |
|   | free_list → [页D-E] → [页F-G] → ...                              |   |
|   | nr_free = 2                                                       |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   free_area[2]：管理所有 4 页的空闲块（order 2）                         |
|   +-------------------------------------------------------------------+   |
|   | free_list → [页H-I-J-K]                                          |   |
|   | nr_free = 1                                                       |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   ...以此类推到 free_area[10]                                             |
|                                                                           |
+===========================================================================+
```

### 2.4 分配过程：一步一步

假设要分配 4 页（order 2）的内存：

```
步骤 1：检查目标阶是否有空闲块
+===========================================================================+
|                                                                           |
|   检查 free_area[2]                                                       |
|   +-------------------------------------------------------------------+   |
|   | free_list → [页H-I-J-K]                                          |   |
|   | nr_free = 1                                                       |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   有空闲块！直接从链表取出，分配成功                                      |
|                                                                           |
+===========================================================================+
```

如果 free_area[2] 为空，需要向上查找：

```
步骤 2：向上查找更大的块
+===========================================================================+
|                                                                           |
|   free_area[2] 为空，检查 free_area[3]                                   |
|   +-------------------------------------------------------------------+   |
|   | free_list → [块A: 8页] → [块B: 8页]                              |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   找到了！但 8 页太大了，需要分裂                                         |
|                                                                           |
+===========================================================================+
```

```
步骤 3：分裂大块
+===========================================================================+
|                                                                           |
|   分裂前：                                                                |
|   +-------------------------------------------------------------------+   |
|   |                    块A: 8 页（order 3）                          |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   分裂后：                                                                |
|   +-------------------------------------------------------------------+   |
|   |       块A左: 4 页       |       块A右: 4 页       |               |   |
|   |       (order 2)         |       (order 2)        |               |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   操作：                                                                  |
|   1. 从 free_area[3] 移除块A                                              |
|   2. 块A左 分配给请求者                                                   |
|   3. 块A右 加入 free_area[2]                                              |
|                                                                           |
+===========================================================================+
```

**内核代码实现**（简化版）：

```c
/*
 * expand 函数：分裂大块
 * 
 * 参数说明：
 *   zone - 内存区域
 *   page - 大块的起始页面
 *   low  - 目标阶数（需要的大小）
 *   high - 当前阶数（找到的大小）
 */
static inline void expand(struct zone *zone, struct page *page,
                          int low, int high, int migratetype)
{
    unsigned long size = 1 << high;  /* 当前块的大小（页数） */

    /*
     * 循环分裂，直到达到目标阶数
     * 
     * 例如：high=3, low=2
     * 第一次循环：high=3, size=8
     *   - high 变成 2
     *   - size 变成 4
     *   - 把后半部分（4页）加入 free_area[2]
     * 循环结束：high == low
     */
    while (high > low) {
        high--;                    /* 阶数减 1 */
        size >>= 1;                /* 大小减半 */
        
        /*
         * 把后半部分加入空闲链表
         * page[size] 是后半部分的起始地址
         * 例如：page 是地址 0，size 是 4
         * page[4] 就是地址 4 开始的 4 页
         */
        add_to_free_list(&page[size], zone, high, migratetype);
        set_buddy_order(&page[size], high);  /* 标记阶数 */
    }
    /* 前半部分（page）用于分配，不需要加入链表 */
}
```

### 2.5 释放过程：合并伙伴

释放内存时，如果伙伴也空闲，就合并成更大的块：

```
释放过程示例：
+===========================================================================+
|                                                                           |
|   初始状态：                                                              |
|   +-------------------------------------------------------------------+   |
|   | 块0: 4页(空闲) | 块1: 4页(占用) | 块2: 4页(空闲) | 块3: 4页(占用) |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   释放块1：                                                               |
|   +-------------------------------------------------------------------+   |
|   | 块0: 4页(空闲) | 块1: 4页(空闲) | 块2: 4页(空闲) | 块3: 4页(占用) |   |
|   +-------------------------------------------------------------------+   |
|           ↓                                                               |
|   检查块1的伙伴（块0）是否空闲？是！                                      |
|           ↓                                                               |
|   +-------------------------------------------------------------------+   |
|   |      块0+1: 8页(空闲)      | 块2: 4页(空闲) | 块3: 4页(占用) |       |   |
|   +-------------------------------------------------------------------+   |
|           ↓                                                               |
|   检查块0+1的伙伴（块2+3）是否空闲？块3不空闲，停止合并                   |
|                                                                           |
+===========================================================================+
```

**内核代码实现**（简化版）：

```c
/*
 * __free_one_page 函数：释放一页并尝试合并
 * 
 * 核心逻辑：找到伙伴 → 检查是否空闲 → 合并 → 递归
 */
static inline void __free_one_page(struct page *page,
                                   unsigned long pfn,
                                   struct zone *zone,
                                   unsigned int order)
{
    unsigned long buddy_pfn;
    struct page *buddy;

    /*
     * 循环尝试合并，直到无法合并
     */
    while (order < MAX_ORDER - 1) {
        
        /* 步骤 1：计算伙伴的页帧号 */
        buddy_pfn = __find_buddy_pfn(pfn, order);
        /*
         * __find_buddy_pfn 的实现：
         * return pfn ^ (1 << order);
         * 
         * 例如：pfn = 0, order = 2
         * buddy_pfn = 0 ^ 4 = 4
         */
        
        /* 步骤 2：获取伙伴的 page 结构 */
        buddy = page + (buddy_pfn - pfn);
        
        /* 步骤 3：检查伙伴是否空闲 */
        if (!page_is_buddy(page, buddy, order))
            break;  /* 伙伴不空闲，停止合并 */
        
        /* 步骤 4：从空闲链表移除伙伴 */
        del_page_from_free_list(buddy, zone, order);
        
        /* 步骤 5：计算合并后的起始地址 */
        pfn = buddy_pfn & pfn;  /* 取较小地址 */
        page = page + (pfn - page_to_pfn(page));
        
        /* 步骤 6：阶数加 1，继续尝试合并 */
        order++;
    }
    
    /* 步骤 7：把合并后的块加入空闲链表 */
    add_to_free_list(page, zone, order);
}
```

## 三、SLUB 分配器：如何实现小块内存管理

### 3.1 为什么需要 SLUB？

假设内核要频繁分配 64 字节的结构体：

```
如果只用伙伴系统：
+===========================================================================+
|                                                                           |
|   请求：分配 64 字节                                                      |
|                                                                           |
|   伙伴系统：最小分配 1 页 = 4096 字节                                     |
|   +-------------------------------------------------------------------+   |
|   |  64 字节使用  |              4032 字节浪费                        |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   问题：                                                                  |
|   1. 内部碎片严重：浪费 98.4%                                             |
|   2. 分配慢：每次都要操作伙伴系统                                         |
|   3. 缓存不友好：每次分配的地址不固定                                     |
|                                                                           |
+===========================================================================+
```

SLUB 的解决方案：**把一页分成多个相同大小的对象**。

```
SLUB 的做法：
+===========================================================================+
|                                                                           |
|   一页（4096 字节）分成 64 字节的对象：                                   |
|   +-------------------------------------------------------------------+   |
|   | obj0 | obj1 | obj2 | obj3 | obj4 | ... | obj63 |                   |   |
|   | 64B  | 64B  | 64B  | 64B  | 64B  |     | 64B  |                    |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   优点：                                                                  |
|   1. 无内部碎片：64 字节刚好用完                                          |
|   2. 分配快：直接从预分配的池中取                                         |
|   3. 缓存友好：相同类型的对象放在一起                                     |
|                                                                           |
+===========================================================================+
```

### 3.2 SLUB 的三层结构

```
SLUB 三层结构：
+===========================================================================+
|                                                                           |
|   第一层：kmem_cache（缓存）                                              |
|   +-------------------+                                                    |
|   | 类似：不同类型的床位 |  task_struct 缓存、inode 缓存等              |
|   | 每种对象一个缓存     |  专门管理一种大小的对象                      |
|   +-------------------+                                                    |
|                                                                           |
|   第二层：slab（页）                                                      |
|   +-------------------+                                                    |
|   | 类似：一个房间       |  一页内存，包含多个对象                      |
|   | 状态：满/部分满/空   |  根据使用情况分类                            |
|   +-------------------+                                                    |
|                                                                           |
|   第三层：object（对象）                                                  |
|   +-------------------+                                                    |
|   | 类似：床位           |  实际分配的内存块                            |
|   | 大小固定             |  从 slab 中分配                              |
|   +-------------------+                                                    |
|                                                                           |
+===========================================================================+
```

### 3.3 数据结构详解

```c
/*
 * kmem_cache 结构 - 管理一种类型的对象
 * 
 * 例如：task_struct 的缓存、inode 的缓存
 */
struct kmem_cache {
    /* 每个 CPU 的本地缓存（快速路径） */
    struct kmem_cache_cpu __percpu *cpu_slab;
    
    /* 全局的 partial 链表（慢速路径） */
    struct list_head partial;
    
    /* 对象大小 */
    unsigned int object_size;  /* 实际对象大小 */
    unsigned int size;         /* 包含元数据的大小 */
    
    /* 向伙伴系统申请的阶数 */
    unsigned int order;        /* 一次申请 2^order 页 */
};

/*
 * kmem_cache_cpu 结构 - CPU 本地缓存
 * 
 * 这是 SLUB 高效的关键：每个 CPU 有自己的缓存，无锁访问
 */
struct kmem_cache_cpu {
    void **freelist;        /* 空闲对象链表 */
    struct page *page;      /* 当前正在使用的 slab */
    struct page *partial;   /* 本地 partial slab 链表 */
};
```

```
数据结构关系：
+===========================================================================+
|                                                                           |
|   kmem_cache（task_struct 缓存）                                         |
|   +-------------------------------------------------------------------+   |
|   | cpu_slab[CPU0] | cpu_slab[CPU1] | cpu_slab[CPU2] | ...             |   |
|   +-------------------------------------------------------------------+   |
|         |                                                                   |
|         v                                                                   |
|   kmem_cache_cpu（CPU0 的本地缓存）                                       |
|   +-------------------------------------------------------------------+   |
|   | freelist → [obj1] → [obj3] → [obj5] → NULL                       |   |
|   | page → 当前 slab                                                  |   |
|   +-------------------------------------------------------------------+   |
|         |                                                                   |
|         v                                                                   |
|   slab（一页内存）                                                         |
|   +-------------------------------------------------------------------+   |
|   | obj0(用) | obj1(空) | obj2(用) | obj3(空) | obj4(用) | obj5(空) |   |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
+===========================================================================+
```

### 3.4 分配过程：快速路径与慢速路径

SLUB 采用两级分配策略：

```
分配策略：
+===========================================================================+
|                                                                           |
|   快速路径（Fast Path）：                                                 |
|   +-------------------+                                                    |
|   | 1. 检查 CPU 本地缓存 freelist                                       |
|   | 2. 如果有空闲对象，直接取出返回                                     |
|   | 3. 无需加锁，速度极快                                               |
|   +-------------------+                                                    |
|                                                                           |
|   慢速路径（Slow Path）：                                                 |
|   +-------------------+                                                    |
|   | 1. 本地缓存为空，需要补充                                           |
|   | 2. 从本地 partial 链表取 slab                                       |
|   | 3. 从全局 partial 链表取 slab                                       |
|   | 4. 向伙伴系统申请新页                                               |
|   +-------------------+                                                    |
|                                                                           |
+===========================================================================+
```

**快速路径代码**（简化版）：

```c
/*
 * 快速路径：从 CPU 本地缓存分配
 * 
 * 这是内联函数，编译后直接嵌入调用处，无函数调用开销
 */
static __always_inline void *slab_alloc(struct kmem_cache *s, gfp_t gfpflags)
{
    struct kmem_cache_cpu *c;
    void *object;
    
    /* 步骤 1：获取当前 CPU 的缓存 */
    c = this_cpu_ptr(s->cpu_slab);
    
    /* 步骤 2：检查本地 freelist 是否有空闲对象 */
    object = c->freelist;
    if (likely(object)) {
        /*
         * 有空闲对象！
         * freelist 是一个链表，每个空闲对象保存下一个空闲对象的地址
         */
        c->freelist = get_freepointer(s, object);  /* 更新 freelist */
        return object;
    }
    
    /* 步骤 3：本地缓存为空，进入慢速路径 */
    return __slab_alloc(s, gfpflags);
}
```

**慢速路径代码**（简化版）：

```c
/*
 * 慢速路径：补充 CPU 本地缓存
 */
static void *___slab_alloc(struct kmem_cache *s, gfp_t gfpflags,
                           struct kmem_cache_cpu *c)
{
    void *freelist;
    struct page *page;
    
    /* 步骤 1：检查当前 slab 是否有对象 */
    page = c->page;
    if (!page)
        goto new_slab;  /* 没有 slab，需要新建 */
    
    /* 步骤 2：尝试从当前 slab 获取对象 */
    freelist = get_freelist(s, page);
    if (freelist)
        goto load_freelist;
    
    /* 步骤 3：当前 slab 已满，检查本地 partial 链表 */
    if (c->partial) {
        page = c->partial;
        c->partial = page->next;
        c->page = page;
        freelist = get_freelist(s, page);
        goto load_freelist;
    }
    
new_slab:
    /* 步骤 4：需要向伙伴系统申请新页 */
    page = allocate_slab(s, gfpflags);
    if (!page)
        return NULL;
    
    c->page = page;
    freelist = page->freelist;
    
load_freelist:
    /* 步骤 5：更新本地缓存 */
    c->freelist = freelist;
    return freelist;
}
```

### 3.5 释放过程：放回本地缓存

释放比分配更简单：直接放回 CPU 本地缓存。

```
释放过程：
+===========================================================================+
|                                                                           |
|   kfree(object) 或 kmem_cache_free(cache, object)                        |
|           |                                                               |
|           v                                                               |
|   +-------------------+                                                    |
|   | 1. 找到对象所属的 slab |  通过对象地址反查 page                      |
|   +-------------------+                                                    |
|           |                                                               |
|           v                                                               |
|   +-------------------+                                                    |
|   | 2. 放入 CPU 本地缓存 |  加入 freelist 链表头                        |
|   +-------------------+                                                    |
|           |                                                               |
|           v                                                               |
|   +-------------------+                                                    |
|   | 3. 检查 slab 是否全空 |                                              |
|   +-------------------+                                                    |
|           |                                                               |
|           +-----+------------------+                                       |
|           | 是                  | 否                                      |
|           v                       v                                       |
|   +-------------------+   +-------------------+                            |
|   | 4a. 释放给伙伴系统 |   | 4b. 保持现状     |                            |
|   +-------------------+   +-------------------+                            |
|                                                                           |
+===========================================================================+
```

**释放代码**（简化版）：

```c
/*
 * 释放对象到 CPU 本地缓存
 */
static __always_inline void slab_free(struct kmem_cache *s, void *object)
{
    struct kmem_cache_cpu *c;
    
    /* 步骤 1：获取当前 CPU 的缓存 */
    c = this_cpu_ptr(s->cpu_slab);
    
    /*
     * 步骤 2：把对象加入 freelist 链表头
     * 
     * freelist 结构：
     * 释放前：freelist → [obj1] → [obj2] → NULL
     * 释放后：freelist → [object] → [obj1] → [obj2] → NULL
     */
    set_freepointer(s, object, c->freelist);
    c->freelist = object;
    
    /* 步骤 3：如果 slab 全空，考虑释放给伙伴系统（延迟释放） */
}
```

### 3.6 freelist 的实现技巧

SLUB 使用了一个巧妙的技巧：**利用对象本身的空间存储链表指针**。

```
freelist 存储方式：
+===========================================================================+
|                                                                           |
|   对象被释放后，空间可以复用：                                            |
|   +-------------------------------------------------------------------+   |
|   | 对象内存区域                                                      |   |
|   +-------------------------------------------------------------------+   |
|   | 正在使用时：存储实际数据                                          |   |
|   | 空闲时：存储下一个空闲对象的地址（指针）                          |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   示例：64 字节对象                                                       |
|   +-------------------------------------------------------------------+   |
|   | [0-7 字节]  | [8-63 字节]                                        |   |
|   | next 指针  | 未使用                                              |   |
|   +-------------------------------------------------------------------+   |
|                                                                           |
|   优点：不需要额外的内存存储链表                                          |
|                                                                           |
+===========================================================================+
```

## 四、伙伴系统与 SLUB 的协作

### 4.1 SLUB 如何向伙伴系统申请内存

当 SLUB 需要新 slab 时，调用伙伴系统：

```c
/*
 * allocate_slab - 向伙伴系统申请页面
 */
static struct page *allocate_slab(struct kmem_cache *s, gfp_t flags)
{
    struct page *page;
    unsigned int order = s->order;  /* 缓存配置的阶数 */
    
    /*
     * 调用伙伴系统分配页面
     * 
     * alloc_pages 会：
     * 1. 找到合适的 zone
     * 2. 从 free_area 中找到空闲块
     * 3. 可能需要分裂大块
     * 4. 返回 page 结构
     */
    page = alloc_pages(flags, order);
    if (!page)
        return NULL;
    
    /*
     * 初始化 slab
     * 把页面分割成多个对象，构建 freelist
     */
    init_slab(page, s);
    
    return page;
}
```

## 五、实际应用示例

### 5.1 伙伴系统使用示例

```c
#include <linux/mm.h>
#include <linux/gfp.h>
#include <linux/module.h>

static int __init buddy_example_init(void)
{
    unsigned long addr;
    struct page *page;
    int order = 2;  /* 申请 4 页 */
    
    pr_info("=== 伙伴系统示例 ===\n");
    
    /*
     * 方法 1：__get_free_pages
     * 返回虚拟地址，适合需要直接访问内存的场景
     */
    addr = __get_free_pages(GFP_KERNEL, order);
    if (!addr) {
        pr_err("分配失败\n");
        return -ENOMEM;
    }
    pr_info("分配成功：虚拟地址 = 0x%lx，大小 = %d 页\n", addr, 1 << order);
    
    /* 使用内存... */
    memset((void *)addr, 0, PAGE_SIZE << order);
    
    /* 释放 */
    free_pages(addr, order);
    pr_info("已释放\n");
    
    /*
     * 方法 2：alloc_pages
     * 返回 page 结构，适合需要操作页面属性的场景
     */
    page = alloc_pages(GFP_KERNEL, order);
    if (!page) {
        pr_err("分配失败\n");
        return -ENOMEM;
    }
    pr_info("分配成功：page = %p，pfn = %lu\n", 
            page, page_to_pfn(page));
    
    /* 获取虚拟地址 */
    addr = (unsigned long)page_address(page);
    pr_info("虚拟地址 = 0x%lx\n", addr);
    
    /* 释放 */
    __free_pages(page, order);
    pr_info("已释放\n");
    
    return 0;
}

module_init(buddy_example_init);
MODULE_LICENSE("GPL");
```

### 5.2 SLUB 分配器使用示例

```c
#include <linux/slab.h>
#include <linux/module.h>

/* 定义一个结构体 */
struct my_struct {
    int id;
    char name[32];
    void *data;
};

static struct kmem_cache *my_cache;

static int __init slub_example_init(void)
{
    struct my_struct *obj1, *obj2, *obj3;
    
    pr_info("=== SLUB 分配器示例 ===\n");
    
    /*
     * 方法 1：kmalloc（通用分配）
     * 内核会自动选择合适的缓存
     */
    obj1 = kmalloc(sizeof(struct my_struct), GFP_KERNEL);
    if (!obj1) {
        pr_err("kmalloc 失败\n");
        return -ENOMEM;
    }
    pr_info("kmalloc 成功：%p，大小 = %zu\n", obj1, sizeof(struct my_struct));
    
    obj1->id = 1;
    strcpy(obj1->name, "test");
    kfree(obj1);
    pr_info("kfree 完成\n");
    
    /*
     * 方法 2：kmem_cache（专用缓存）
     * 为特定类型创建专用缓存，性能更好
     */
    my_cache = kmem_cache_create("my_struct_cache",
                                  sizeof(struct my_struct),
                                  0, 0, NULL);
    if (!my_cache) {
        pr_err("创建缓存失败\n");
        return -ENOMEM;
    }
    pr_info("创建专用缓存成功\n");
    
    /* 从专用缓存分配 */
    obj2 = kmem_cache_alloc(my_cache, GFP_KERNEL);
    obj3 = kmem_cache_alloc(my_cache, GFP_KERNEL);
    
    pr_info("分配对象：%p, %p\n", obj2, obj3);
    
    /* 释放回专用缓存 */
    kmem_cache_free(my_cache, obj2);
    kmem_cache_free(my_cache, obj3);
    pr_info("释放完成\n");
    
    /* 销毁缓存 */
    kmem_cache_destroy(my_cache);
    pr_info("销毁缓存完成\n");
    
    return 0;
}

module_init(slub_example_init);
MODULE_LICENSE("GPL");
```

## 六、总结

### 6.1 伙伴系统要点

| 方面 | 说明 |
|------|------|
| **核心思想** | 按 2 的幂次方划分内存 |
| **最小单位** | 1 页（4KB） |
| **最大单位** | 1024 页（4MB） |
| **分配策略** | 找目标阶 → 没有则向上找 → 分裂大块 |
| **释放策略** | 找伙伴 → 伙伴空闲则合并 → 递归合并 |
| **优点** | 减少外部碎片，分配连续物理内存 |
| **缺点** | 内部碎片（最小 4KB），不适合小对象 |

### 6.2 SLUB 分配器要点

| 方面 | 说明 |
|------|------|
| **核心思想** | 把页面分割成固定大小的对象 |
| **三层结构** | kmem_cache → slab → object |
| **快速路径** | 从 CPU 本地缓存分配，无锁 |
| **慢速路径** | 从 partial 链表或伙伴系统获取 |
| **释放策略** | 放回 CPU 本地缓存，全空则释放给伙伴系统 |
| **优点** | 无内部碎片，分配快，缓存友好 |
| **缺点** | 只适合内核小对象，不适合大块内存 |

### 6.3 两者协作关系

```
协作关系：
+===========================================================================+
|                                                                           |
|   应用层请求                                                              |
|       |                                                                   |
|       v                                                                   |
|   +-------------------+                                                    |
|   | 判断大小         |                                                    |
|   +-------------------+                                                    |
|       |                                                                   |
|       +-----+------------------+                                           |
|       | < 1 页              | >= 1 页                                     |
|       v                     v                                             |
|   +-------------------+ +-------------------+                              |
|   | SLUB 分配器      | | 伙伴系统         |                              |
|   | kmalloc/kfree    | | alloc_pages      |                              |
|   +-------------------+ +-------------------+                              |
|       |                     ^                                             |
|       | 需要新 slab         |                                             |
|       +---------------------+                                             |
|                                                                           |
+===========================================================================+
```

伙伴系统管理大块连续物理内存，SLUB 分配器管理小块内核对象，两者协作构成了 Linux 内核高效的内存管理体系。
